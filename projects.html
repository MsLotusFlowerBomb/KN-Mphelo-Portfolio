UL Optometry Clinic Management System
★ Flagship
Cross-platform clinical management system for the University of Limpopo's Optometry Department — Cyber6ix team, LDIL Final Project
C# · .NET MAUI Supabase PostgreSQL MVVM RBAC Row-Level Security JWT Auth GitHub Actions
★ Presented to the Parliamentary Portfolio Committee on Communications & Digital Technologies · April 2026
Overview

A fully digitalised clinical management platform replacing a paper-based workflow at UL's Optometry teaching clinic. Four user roles — Patient, Student, Supervisor, Admin — each with isolated role shells, dedicated dashboards and enforced data boundaries at the database layer.

Developed by the Cyber6ix team as the LDIL final project, it was presented to the Parliamentary Portfolio Committee on Communications and Digital Technologies in April 2026 — stakeholder validation at the highest institutional level in South Africa.

Tech Stack
Frontend.NET MAUI (C#) — cross-platform: Android, iOS, Windows, macOS
BackendSupabase (PostgreSQL + PostgREST API + Realtime)
ArchitectureMVVM + Service Layer (CommunityToolkit.Mvvm)
AuthSupabase Auth — JWT (30-min expiry) + refresh tokens
SecurityPostgreSQL Row-Level Security — data isolation at DB layer
CI/CDGitHub Actions multi-platform build pipeline
Architecture — 4-Tier MVVM + Service Layer
LayerTechnologyResponsibility
ViewsXAML (.NET MAUI)Role-specific screens — Admin, Student, Supervisor, Patient shells
ViewModelsMVVM ToolkitObservableProperty + RelayCommand — UI state, no business logic
Services (×14)C# interfaces + implementationsAuthService, BookingService, EncounterService, ReviewService, PoeService, NotificationService, AuditService, ReportService, SchedulingService…
DataSupabase PostgREST → PostgreSQL15 core tables, RLS policies per role, real-time subscriptions
Key Features
  • 6-stage booking state machine: Patient books → Auto-confirm → Student accepts → Admin assigns supervisor → Encounter filled → Supervisor signs off
  • 6-step clinical encounter wizard — VA, Refraction, NPC, Cover Test, Anterior/Posterior Segment, Reflection, Supervisor review
  • Portfolio of Evidence (PoE) — 5 categories, 500 required hours, auto-credited per approved encounter
  • Notification routing — real-time push per role on every state change
  • Audit log — every user action timestamped and stored, HIPAA-equivalent handling
  • Blocked dates — admin-controlled calendar for clinic closures
Security Design
  • PostgreSQL RLS — a compromised client token cannot read unauthorised rows, enforced at DB layer not application layer
  • JWT tokens (30-min expiry) + refresh token rotation
  • HIPAA-equivalent PII handling — patient data encrypted and audit-logged
  • Role-Based Access Control: Admin > Supervisor > Student > Patient — enforced in both ViewModel guards and RLS policies
  • Input validation on all encounter form fields before submission
Reflections
  • PostgreSQL RLS as the security boundary is architecturally superior to application-layer guards — even a fully compromised client cannot breach data isolation.
  • Designing four role shells simultaneously required strict interface segregation from the first commit — retrofitting it later would have been costly.
  • Presenting to the Parliamentary Portfolio Committee taught that a system's value must be argued in outcomes (patient safety, time saved) not architecture — a lesson that now shapes how I scope every project.

TenderShield SA
Cybersecurity · AI
AI-powered procurement fraud detection platform — itweb Security Summit Hackathon
ReactClaude Sonnet API SupabaseVercel Upstash RedisPOPIA PFMAB-BBEEPPPFA
Overview

TenderShield SA detects and flags fraudulent patterns in South African government procurement tenders. Built in the itweb Security Summit hackathon, it integrates the Claude Sonnet API for live AI fraud analysis against a curated SA regulatory knowledge base.

South Africa loses billions annually to procurement fraud. TenderShield contextualises suspicious patterns against seven Acts and frameworks simultaneously — giving analysts actionable flagging with specific clause references.

Regulatory Knowledge Base
PPPFAPreferential Procurement Policy Framework Act — scoring and preference points
PFMAPublic Finance Management Act — national/provincial spending controls
MFMAMunicipal Finance Management Act — municipal procurement governance
B-BBEEBroad-Based Black Economic Empowerment — transformation compliance
CIDBConstruction Industry Development Board — contractor grading
CSDCentral Supplier Database — vendor registration verification
POPIAProtection of Personal Information Act — data privacy
AI Fraud Analysis Pipeline
  • Vendor B-BBEE compliance and CSD registration verification
  • Price anomaly detection relative to market benchmarks — potential price-fixing signals
  • Sole-source justification validation against PPPFA thresholds
  • Related-party relationship detection between evaluators and bidders
  • Irregular award pattern detection — same vendor winning across departments
  • Risk score output: Low / Medium / High / Critical with clause references
Tech Stack
FrontendReact — dark cybersecurity aesthetic, red/dark palette
AIClaude Sonnet API — structured prompt chain per regulatory framework
DatabaseSupabase (PostgreSQL + Auth)
CacheUpstash Redis — rate limiting + API response caching
HostingVercel
StorageAWS S3 — document storage
DataApify — tender data ingestion pipeline
Reflections
  • Regulatory research (seven Acts across national, provincial, and municipal tiers) was as technically demanding as the code — domain knowledge is a genuine engineering asset in civic-tech.
  • LLM fraud detection requires explicit regulatory grounding in the system prompt — generic analysis without SA-specific context produces irrelevant output.
  • The dark cybersecurity aesthetic was a deliberate UX choice — security tooling's visual language signals authority and seriousness, which directly affects analyst trust in the output.

StealthMode — Static Context Engine (SCE)
AI · Operational Security
Offline AI coding assistant engineered for restricted university lab environments — zero network footprint, USB-portable, course-specific knowledge base
JavaGemini API LangChain4jApache Lucene GlassFishNetBeans Ant ADB Tunnel
Problem Statement
EnvironmentUniversity lab PCs — NetBeans + GlassFish + Derby only; no Maven downloads; no admin rights
NetworkRestricted or monitored institutional network; external API calls logged
VisibilityNo suspicious plugins or installations visible to other users or staff
PortabilityEntire system must fit on a USB drive or single JAR file
IntelligenceMust generate course-specific code — EJBs, Servlets, JDBC patterns, Memani's exact Interface+Manager architecture
Architecture Evolution
  • Phase 1 — USB Dev Environment: Full portable NetBeans + JDK + Maven repo on USB; netbeans.conf redirected all cache dirs to USB path — zero host trace
  • Phase 2 — Fat JAR: Single pre-compiled JAR (LangChain4j + Gson + Lucene bundled, named derby-driver-addon-ext.jar) in WEB-INF/lib — indistinguishable from a standard library
  • Phase 3 — Final SCE: Gemini API-powered Ant project; course content baked in; CodeTemplateBroker class maps scenario × layer → boilerplate; API routed via ADB USB tunnel (phone as cellular gateway)
Core Components
CodeTemplateBrokerOffline code dictionary — scenario + layer → pre-formatted boilerplate for 8 academic scenarios × 4 layers
Lucene IndexApache Lucene indexes knowledge base .txt files on USB — keyword search maps academic questions to code segments
LangChain4j StoreInMemoryEmbeddingStore — semantic retrieval of past templates and JavaDoc, no persistence required
Stealth ServletPerformanceTrackerServlet — disguised as a system diagnostic tool; renders AI UI inside a standard-looking admin page
ADB TunnelPhone tethered via USB → ADB reverse tunnel → Gemini API calls bypass institutional network entirely
Operational Security Design
JAR namingderby-driver-addon-ext.jar — indistinguishable from standard Derby libs
Servlet nameSystemPerformanceMonitor, URL mapped to /sys-perf-trace
DeploymentCopy JAR → GlassFish deploys → unplug USB; server reads from domain1/applications/ cache
NetworkZero host-network traffic — all API calls via phone's cellular connection over ADB USB
CleanupRemove servlet source → Clean + Build in NetBeans → clear Recent Projects history before submission
Reflections
  • This is primarily a systems design and operational security exercise — the interesting engineering challenge is constraint-satisfaction, not the AI component itself.
  • The CodeTemplateBroker pattern (compiled Java class as an offline knowledge dictionary) is broadly applicable to any constrained environment where an LLM is unavailable.
  • The ADB reverse tunnel solution — exploiting the fact that USB device traffic bypasses the institutional network stack — is a genuinely novel approach to network restriction.

StudentResFinder
Java EE · Security
Student accommodation discovery platform — TUT group project. My role: Security & Admin module
Java EEApache Derby JSTLGlassFish bcryptRBAC Apache Ant
Overview

A Java EE platform connecting students seeking accommodation with verified landlords. Three-role architecture — Student, Landlord, Admin — with a landlord verification workflow, listing management, and student-facing search and filter.

My contribution: the Security & Admin module — implementing the admin verification workflow, GlassFish realm configuration, and security utilities used across the entire application by all team members.

Security Contributions
  • GlassFish file realm with STUDENT, LANDLORD, ADMIN groups — group-to-role mapping in glassfish-web.xml aligned with web.xml security constraints
  • SecurityUtil.requireRole() — programmatic role enforcement consumed by every servlet in the project, clean interface contract from day one
  • PasswordUtil — bcrypt password hashing with salt; zero plaintext storage
  • HTTP-only session cookies configured in web.xml
  • Input sanitisation helpers and output escaping across JSPs
Admin Verification Workflow

Landlords register with status PENDING — blocked from creating listings until verified. Admin dashboard surfaces all pending accounts. Verify or Reject action triggers VerificationSB → LandlordDAO → status update. Rejected landlords cannot publish accommodation. This workflow was designed so that other team members could call it with a single method — abstraction that enabled parallel development across the group project.
